# エスカレーション基準

## 優先度定義

| 優先度 | 名称 | 基準 | 初回応答SLA | 解決SLA |
|--------|------|------|-----------|---------|
| P1 | 緊急 | サービス全面停止、データ損失の恐れ | 30分以内 | 4時間以内 |
| P2 | 高 | 主要機能の障害、多数ユーザーに影響 | 2時間以内 | 8時間以内 |
| P3 | 中 | 一部機能の不具合、ワークアラウンドあり | 4時間以内 | 3営業日以内 |
| P4 | 低 | 軽微な問題、UI改善要望、一般質問 | 1営業日以内 | 5営業日以内 |

## エスカレーション判断フロー

```
問い合わせ受付
  ├── データ損失・セキュリティインシデント？ → 即時エスカレーション（P1 + セキュリティチーム）
  ├── サービス停止・全ユーザー影響？ → P1 → インフラチーム + マネージャー通知
  ├── 特定顧客の業務停止？ → P2 → テクニカルサポート
  ├── 機能不具合だがワークアラウンドあり？ → P3 → 通常対応
  └── 質問・要望 → P4 → 1次対応で完結
```

## 即時エスカレーションが必要なケース

以下に該当する場合は、**担当者判断を待たずに即座にエスカレーション**してください。

1. **セキュリティインシデント**: 不正アクセス、情報漏洩の疑い → セキュリティチーム + CISO
2. **法的リスク**: 訴訟示唆、弁護士からの連絡 → 法務部 + 部長
3. **メディア関連**: SNS拡散、報道機関からの問い合わせ → 広報部 + 経営層
4. **VIP顧客**: エンタープライズプラン（年間1,000万円以上）の顧客 → 専任チーム
5. **人身事故**: 製品起因の怪我等 → 品質管理部 + 法務 + 経営層

## エスカレーション先

| チーム | 担当範囲 | 連絡方法 |
|--------|----------|----------|
| L1サポート | 一般問い合わせ、FAQ案内 | チケット |
| L2テクニカル | 技術的調査、ログ解析 | チケット + Slack #support-l2 |
| インフラチーム | サーバー・ネットワーク障害 | Slack #infra-alert + PagerDuty |
| セキュリティチーム | セキュリティインシデント | Slack #security-incident + 緊急電話 |
| 開発チーム | バグ修正、緊急パッチ | Jira + Slack #dev-escalation |
| カスタマーサクセス | 解約防止、大口顧客対応 | 直接連絡 |

## 対応記録の必須項目

すべてのエスカレーションで以下を記録してください。

- 顧客名・契約プラン
- 問い合わせ内容の要約
- 影響範囲（ユーザー数、業務影響度）
- エスカレーション理由
- 現在のステータス
- 次のアクション・期限
