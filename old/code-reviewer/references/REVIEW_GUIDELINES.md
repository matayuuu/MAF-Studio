# コードレビュー ガイドライン

## チェックリスト

### 1. 可読性
- [ ] 変数名・関数名は意図を明確に表現しているか
- [ ] コメントは「なぜ」を説明しているか（「何を」ではなく）
- [ ] 1つの関数は1つの責務に集中しているか
- [ ] ネストが深すぎないか（3段以上は要リファクタリング）

### 2. セキュリティ
- [ ] ユーザー入力のバリデーションを行っているか
- [ ] SQL インジェクション対策（パラメータバインディング）
- [ ] XSS 対策（出力エスケープ）
- [ ] 機密情報（API キー等）がハードコードされていないか
- [ ] 適切な認証・認可チェックが実装されているか

### 3. パフォーマンス
- [ ] N+1 クエリが発生していないか
- [ ] 不要なループ処理がないか
- [ ] キャッシュ可能なデータを毎回取得していないか
- [ ] 大量データの処理にページネーションを使用しているか

### 4. エラーハンドリング
- [ ] 例外を適切にキャッチし、意味のあるメッセージを返しているか
- [ ] リソース（ファイル、接続）を確実にクローズしているか
- [ ] 想定外の入力に対する防御的プログラミング

### 5. テスト
- [ ] ユニットテストが十分にカバーしているか
- [ ] エッジケースのテストがあるか
- [ ] テストが独立して実行可能か

## レビューコメントの書き方

| 種別 | プレフィックス | 例 |
|------|--------------|-----|
| 必須修正 | `[MUST]` | `[MUST] SQL インジェクションの脆弱性があります` |
| 推奨 | `[SHOULD]` | `[SHOULD] この関数は分割した方が読みやすくなります` |
| 提案 | `[NIT]` | `[NIT] 変数名を items → order_items に変更すると意図が明確です` |
| 質問 | `[Q]` | `[Q] このロジックの意図を教えてください` |
